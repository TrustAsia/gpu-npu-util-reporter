# 产品需求文档 (PRD)：GPU/NPU 多源利用率监控与报表生成系统

## 1. 产品概述

本产品是一款基于 Rust 开发的轻量级监控与报表生成工具。其主要目标是从多个数据源的 Prometheus 服务器中提取指定时间范围内的 GPU（NVIDIA A10）和 NPU（Ascend 910B）计算资产使用情况，整合设备占用者信息（如 Container、Pod、Namespace），并根据外部资产表进行数据扩展映射，最终输出排版精美的 Excel 分析报告。

## 2. 核心功能需求

### 2.1 数据源与时间范围配置

* **多源 Prometheus 获取**：支持在配置文件中配置多个 Prometheus server 的连接信息。
* **时间范围动态查询**：
* 允许用户指定数据的提取时间范围。
* 输入格式至少需要支持 `YYYY-MM-DD hh:mm:ss`。


* **自定义设备类型映射**：
* 允许用户在配置中自定义 `device_type` 的底层映射关系（如定义“NVIDIA A10”对应哪些特定标签或特征指标，NPU同理，比如主机IP/容器名称/节点名称/命名空间等）。



### 2.2 设备指标提取规则

系统需要通过 PromQL 查询计算卡的“平均利用率”、“峰值利用率”及“峰值出现时间”。指标提取需兼容以下两类设备：

* **NVIDIA A10 (基于 DCGM Exporter)**
*
**核心利用率**：对应指标 `DCGM_FI_DEV_GPU_UTIL` 。


*
**显存占用率**：需通过组合查询计算显存占用率。


*
**获取已用显存**: `DCGM_FI_DEV_FB_USED` 。


*
**获取空闲显存**: `DCGM_FI_DEV_FB_FREE` 。


*
**显存占用率公式**：`DCGM_FI_DEV_FB_USED / (DCGM_FI_DEV_FB_USED + DCGM_FI_DEV_FB_FREE) * 100` 。


*
**卡编号**：通过 Prometheus 标签中的 `gpu` 字段提取。


* **Ascend 910B (基于 NPU Exporter)**
*
**核心利用率**：对应指标 `npu_chip_info_utilization` 。


*
**显存占用率**：优先查询 `npu_chip_info_hbm_utilization` 。


*
**HBM 降级计算策略**：当无法查到 `npu_chip_info_hbm_utilization` 时，系统必须自动 fallback，使用 `npu_chip_info_hbm_used_memory` 除以 `npu_chip_info_hbm_total_memory` 来计算得出 。


*
**卡编号**：通过标签中的 `id` 字段获取 NPU ID 。




### 2.3 动态资产表映射机制 (Asset Mapping)

为了丰富报表维度，系统需支持灵活的资产信息关联：

* **启用开关**：允许用户全局开启或关闭资产映射功能。
* **数据源定义**：支持配置外部映射源（如本地 CSV/Excel 文件路径）。
* **匹配键 (Mapping Key)**：配置用于关联的基准指标（例如通过 IP 匹配，或通过 “IP + 计算卡编号” 联合匹配）。
* **提取与注入规则 (支持多项)**：
* **提取字段**：可多选外部映射表中的字段（如：机房位置、负责人、部门等）。
* **重命名映射**：允许为提取的列指定在最终 Excel 报告中的新“列名”。
* **动态插入位置**：针对每个映射列，可配置其在 Excel 中的位置（例如：“添加到【主机 IP】列前” 或 “添加到【计算卡编号】列后”）。
  * **锚点约束**：位置锚点（`before`/`after` 引用的列名）**必须是基础报表列之一**（见 §3 基础列清单），不允许以其它映射列作为锚点。若引用的锚点不在基础列中，记 Warning 并把该映射列追加到所有列末尾。



### 2.4 标签与归属取值策略

对于 Kubernetes 等环境中的动态资源分配（Pod/Container/Namespace 等），由于其生命周期可能短于统计的时间范围，系统需提供以下处理策略供用户选择：

* **瞬时值模式 (Current Instant)**：直接拉取当前执行查询时的瞬时标签值作为总结。
* **末态值模式 (Last Value in Range)**：遍历用户指定的时间范围，提取该时间段内该设备最后一个关联的 Container/Pod/Namespace 信息作为总结。



### 2.5 数据处理与统计

对于时间范围内的时序数据，程序需在内存中进行聚合计算：

* **核心/显存利用率平均值**：时间范围内所有数据点的算术平均。
* **核心/显存利用率峰值**：时间范围内的最大值。
* **峰值出现时间**：最大值对应的时间戳（转换为可读的 `YYYY-MM-DD hh:mm:ss` 格式）。

### 2.6 阈值触发与报表染色 (Threshold Highlighting)

为帮助用户快速识别异常或闲置的计算卡，系统需支持基于阈值规则的报表单元格染色功能。

* **触发器类型 (共 8 个)**：针对核心/显存的均值/峰值，分别提供“高于”与“低于”两种比较方向：

| 触发器 | 比较对象 | 比较方向 |
|--------|----------|----------|
| `core_avg_above` | 核心利用率平均值 | 高于阈值 |
| `core_avg_below` | 核心利用率平均值 | 低于阈值 |
| `core_peak_above` | 核心利用率峰值 | 高于阈值 |
| `core_peak_below` | 核心利用率峰值 | 低于阈值 |
| `mem_avg_above` | 显存占用率平均值 | 高于阈值 |
| `mem_avg_below` | 显存占用率平均值 | 低于阈值 |
| `mem_peak_above` | 显存占用率峰值 | 高于阈值 |
| `mem_peak_below` | 显存占用率峰值 | 低于阈值 |

* **启用控制**：每个触发器均可独立启用/禁用（`enabled` 开关）。用户可任意组合启用的触发器集合；未启用的触发器不参与染色判断。
* **阈值单位**：阈值为 0–100 的数值（与利用率原始范围一致，报表显示时再统一按百分比格式化）。
* **染色粒度**：仅对命中触发器的**单个数值单元格**染色（例如 `core_avg_above` 命中时，只染“核心利用率平均值”单元格，不波及整行或其它列）。
* **颜色配置**：每个触发器独立配置自己的染色颜色，颜色必须支持 **HEX 格式**（如 `#FF0000`）。不同触发器可配置不同颜色，便于区分“过载”与“闲置”等不同语义。
* **多触发器叠加**：同一单元格可能同时命中多个触发器（如均值既高于某值又被另一规则覆盖），此时以配置中**最先列出且命中**的触发器颜色为准，避免颜色冲突。
* **N/A 处理**：当某单元格无有效数据（标记为 N/A）时，不参与任何染色判断。
* **默认行为**：开箱即用生成的默认配置中，所有触发器均为 `enabled: false`，即默认不染色；用户显式开启并填写阈值后方才生效。

## 3. 报表输出与排版 (Excel)

系统将生成 `.xlsx` 格式的报表，列必须包含但不限于以下基础项（以及配置的资产映射项）：

* **基础信息**：数据来源（Prometheus URL/别名）、主机 IP、节点名称、计算卡编号、设备类型。
*
**占用来源**：Namespace、Pod、容器名称。


* **统计参数**：取值时间范围。
* **核心指标**：核心利用率平均值、核心利用率峰值、核心利用率峰值出现时间。
* **显存指标**：显存占用率平均值、显存占用率峰值、显存占用率峰值出现时间。

**外观格式排版要求：**

* 标题栏需冻结首行，应用加粗和背景色。
* 数值列（如利用率）需格式化为百分比（保留两位小数，添加百分号）。
* 自动调整列宽以适应内容（如 IP 和时间戳字段）。
* **阈值染色**：对启用的阈值触发器（见 §2.6），将命中条件的利用率数值单元格填充为该触发器配置的 HEX 颜色背景，便于在报表中一眼定位过载/闲置的计算卡。

## 4. 配置文件要求 (YAML)

* 必须使用 YAML 作为配置承载载体。
* **开箱即用机制**：应用启动时，若检测到指定路径下无配置文件，必须自动在当前目录生成包含所有支持项、且带有详细中文注释的默认 `config.yaml`。

## 5. 非功能性需求与架构设计

### 5.1 高内聚低耦合的架构划分 (Rust)

程序结构需遵循清晰的模块化设计：

* **配置模块 (`config`)**：负责 YAML 的解析、反序列化（基于 `serde`）以及默认配置文件的生成。
* **数据源适配层 (`fetcher`)**：通过 Trait (如 `MetricFetcher`) 抽象不同 Exporter 的查询逻辑，解耦 DCGM 和 NPU 的具体 PromQL 拼装。
* **数据处理与聚合 (`processor`)**：负责时间范围内的峰值/均值计算，以及 HBM fallback 计算逻辑的执行。
* **资产映射引擎 (`mapper`)**：独立处理资产映射的 Join 逻辑和列位置排布算法。
* **报告渲染层 (`reporter`)**：隔离底层的 Excel 写入库，专职处理数据到表格的转换与排版样式设定。

### 5.2 错误提示与健壮性

* 使用 Rust 标准的 `Result` 和强类型的错误传播机制（推荐使用 `anyhow` 或 `thiserror`）。
* **终端提示**：对于网络超时、PromQL 语法错误、JSON 解析失败、配置文件读写异常等，必须在控制台输出带有上下文的、对人类友好的中文错误提示（例如：“[错误] 无法连接到配置的 Prometheus 数据源 192.168.1.100，请检查网络或配置” 而不是直接 panic 抛出网络底层异常）。
* 对于单张卡或单一 Prometheus 数据源查询失败，程序应记录 Warning 并在最终报表中对应字段留空或标记 "N/A"，而不应该引起整个程序崩溃（Panic）。

### 5.3 代码规范

* 所有 Rust 源码中的 `struct`、`trait`、复杂算法块以及配置文件，均须包含完善详细的中文注释，说明其设计意图和字段含义。
